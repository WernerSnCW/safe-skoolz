### CHANGE 5 — FIX-02: Locked pupils endpoint

File: artifacts/api-server/src/routes/auth.ts (or schools.ts — see
decision below)

DECISION REQUIRED: First read POST /pupils/reset-pin/:id in schools.ts.
If it already sets lockedUntil = null and failedLoginAttempts = 0:
  - Use that endpoint as the unlock action from the UI
  - No new unlock endpoint needed
  - State "Using existing reset-pin endpoint" in the build report

If it does NOT clear lockedUntil and failedLoginAttempts:
  - Add a new endpoint POST /api/auth/unlock-pupil/:id
  - Auth: requireRole("coordinator", "head_teacher")
  - Sets lockedUntil = null, failedLoginAttempts = 0
  - Writes audit log: eventType "pupil_lockout_cleared"
  - State "New unlock endpoint created" in the build report

Regardless of which path is taken, add:

GET /api/auth/locked-pupils
- Auth: requireRole("coordinator", "head_teacher")
- Query: usersTable WHERE schoolId = user.schoolId AND role = "pupil"
  AND lockedUntil > new Date()
- Returns: array of { id, firstName, lastName, className, yearGroup,
  lockedUntil, failedLoginAttempts }
- Returns empty array (not 404) when no pupils are locked
- Ordered by lockedUntil desc

CRITICAL: declare this static GET route BEFORE any existing parameterised
/:id routes in the same router file.

Mount in routes/index.ts if adding to a new file. If adding to auth.ts,
it is already mounted.
