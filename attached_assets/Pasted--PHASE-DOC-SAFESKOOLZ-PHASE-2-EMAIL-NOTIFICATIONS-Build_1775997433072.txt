## STEP 2 SIGN-OFF

**Agreed idea (one paragraph):**
Add email notifications for three high-priority safeguarding events that
currently only produce in-app notifications. The three triggers are:
(1) a parent receives an incident disclosure — email sent to the parent's
email address; (2) a new Tier 2 or Tier 3 incident is created — email
sent to the school's coordinator(s) and head_teacher(s); (3) a pattern
alert is created at Amber or Red severity — email sent to coordinator(s)
and head_teacher(s). All three use the same email delivery infrastructure.
Replit AI must scan for existing email packages before choosing a
provider — the gate from the PDF build applies here. Email sending is
non-blocking: if the email fails, the primary operation (incident
creation, pattern alert creation, disclosure) still succeeds. A failed
email send is logged to the audit trail but does not return an error to
the caller.

**Open assumptions going into Step 3:**
- No email infrastructure currently exists — confirmed from Step 1 scan
  (no SendGrid, Resend, or Nodemailer packages found). Replit AI must
  scan package.json files across the monorepo to confirm before choosing
  a provider.
- Coordinator and head_teacher email addresses exist in usersTable —
  confirmed from data model (email field nullable). Replit AI must verify
  that seeded coordinator/head_teacher accounts have non-null emails
  before implementing recipient lookup.
- Parent email addresses exist in usersTable — same check required.
- The disclosure endpoint in incidents.ts triggers when a disclosure
  request is responded to with "approved" — Replit AI to confirm the
  exact location in incidents.ts before adding email send.
- Pattern alert creation in patternDetection.ts uses a direct DB insert —
  Replit AI to confirm this is the single place to hook into.
- Incident creation tier check in incidents.ts — Replit AI to confirm
  escalationTier is set at creation time and is available in the POST
  /incidents handler.

### New file: artifacts/api-server/src/lib/emailHelper.ts

```
sendEmail({
  to: string,           // recipient email address
  toName: string,       // recipient display name (for personalisation)
  subject: string,
  bodyText: string,     // plain text body
  trigger: string,      // event type — for audit log on failure
  recipientId: string,  // userId — for audit log on failure
  schoolId: string,     // for audit log on failure
}): Promise<void>
```

Behaviour:
- Calls the chosen provider's send function
- FROM address: process.env.EMAIL_FROM_ADDRESS ?? "noreply@safeskoolz.com"
- If send succeeds: return silently
- If send throws or returns an error: catch, call writeAudit() with
  eventType "email_send_failed", details: { trigger, recipientId,
  error: error.message }, then return silently (do not rethrow)
- Never throws — caller is never interrupted by email failures

### Email trigger 1: Incident disclosure approved → parent email

**File:** artifacts/api-server/src/routes/incidents.ts

Find the disclosure response endpoint (where status is set to "approved").
After the DB update succeeds and after any existing in-app notification,
add a non-blocking email send:

Recipient: the parent user (look up by disclosure.parentId →
usersTable to get email and firstName)

Subject: "Safeskoolz — Incident information shared with you"

Body (plain text):
```
Dear [parent firstName],

The school has shared information about a safeguarding incident
([referenceNumber]) with you.

Please log in to Safeskoolz to view the details and acknowledge receipt.

If you have any concerns, please contact the school directly.

This message was sent by [school name] via Safeskoolz.
This email is confidential. Please do not forward it.
```

If parent has no email address: skip send silently (log to console,
do not audit — not an error condition).

### Email trigger 2: New Tier 2 or Tier 3 incident → coordinator email

**File:** artifacts/api-server/src/routes/incidents.ts

Find the POST /incidents handler. After the incident is inserted and
escalationTier is determined, if escalationTier >= 2, after any existing
in-app notifications, add a non-blocking email send to all active
coordinator and head_teacher users in the school:

Look up recipients: usersTable WHERE schoolId = school AND role IN
("coordinator", "head_teacher") AND active = true AND email IS NOT NULL

Subject: "Safeskoolz — [Tier 2/Tier 3] Incident Reported: [referenceNumber]"

Body (plain text):
```
A [Tier 2 / Tier 3] safeguarding incident has been reported at [school name].

Reference: [referenceNumber]
Category: [category]
Reported: [date and time]

Please log in to Safeskoolz to review and assess this incident.

This is an automated alert from Safeskoolz.
```

If no coordinator/head_teacher has an email address: skip silently.
Send one email per recipient (not a single bulk email).
Each send is independent — one failure does not block others.

## CONSTRAINTS

- Provider gate must be completed and stated before any implementation
- emailHelper.ts must never throw — all errors caught internally
- Email sends are always non-blocking — do not await in a way that
  delays the HTTP response to the caller
- Implement as: after res.json() or after the primary operation, fire
  email sends with Promise.all(...).catch(() => {}) or equivalent
- No HTML emails — plain text only in this build
- EMAIL_FROM_ADDRESS must be read from env var with fallback
- One new env var for provider API key — name must match the provider:
  RESEND_API_KEY, SENDGRID_API_KEY, etc.
- Both env vars must be added to .replit or documented in the build
  report as required env vars
- No schema changes
- writeAudit() on email failure only — do not audit successful sends
  (would create excessive log noise)
- Pattern detection runs on a schedule — emailHelper must be importable
  from patternDetection.ts without circular dependency issues. Verify
  import path before implementing.
- Backward compatibility: existing in-app notifications are unchanged —
  email is additive
