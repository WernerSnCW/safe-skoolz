## Options for next phase

**Option A — FIX-04: In-memory state / autoscale hardening (HIGH effort)**
Addresses the three in-memory state weaknesses that create reliability
issues in Replit's autoscale deployment: pupil login sessions lost on
restart, duplicate pattern detection alerts, ineffective rate limiting.
Requires PostgreSQL session store, DB-backed scan lock, pg rate limit store.
This is the highest-impact remaining technical item.
Step 1 required: YES — scope to auth.ts, index.ts, app.ts, and any
session/rate limit middleware before speccing.

**Option B — FIX-05: API pattern consolidation (MEDIUM effort)**
Migrate remaining raw fetch() calls in diary.tsx, behaviour.tsx,
my-class.tsx, diagnostics.tsx to generated React Query hooks. Reduces
maintenance burden and standardises error handling across the frontend.
Low risk, no backend changes.
Step 1 required: YES — scope to the four frontend files above to confirm
current fetch patterns before migrating.

**Option C — FIX-11: Protocol progressive disclosure (MEDIUM effort)**
Phase the protocol detail page by status — show minimal fields when
status is "open", progressively reveal interviews, referrals, and
closure sections as status advances. UX improvement for coordinators
managing early-stage protocols.
Step 1 required: YES — scope to protocols/detail.tsx to map current
sections and status values before restructuring.

**Option D — New feature: Email notifications**
The notification system currently creates in-app notifications only.
Adding email delivery (SendGrid or similar) for high-priority events
(Help Now, diary scan skipped, new disclosure) would close a real gap
for coordinators who may not check the platform constantly.
Step 1 required: YES — full codebase scan to confirm no email
infrastructure exists before speccing.
