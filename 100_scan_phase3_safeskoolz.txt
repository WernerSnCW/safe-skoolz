### 3. Data flow

- **Incident creation flow:**
  1. Client POST /incidents with auth token → auth middleware extracts
     JwtPayload
  2. Body parsed with CreateIncidentBody Zod schema
  3. `determineEscalationTier(category)` → tier 1/2/3
  4. `isSafeguardingTrigger(category)` → boolean
  5. `generateIncidentRef()` → reference number (SS-YYYY-NNNN)
  6. DB insert into incidentsTable → returning incident record
  7. `writeAudit()` → audit_log row (eventType "incident_created")
  8. In-app notification to reporter (if not anonymous)
  9. In-app notifications to coordinators + head_teachers (if tier >= 3
     or safeguardingTrigger)
  10. `runPatternDetection(incident)` → fire-and-forget (pattern analysis
      + potential alert creation + potential email)
  11. `buildProtocolGuidance()` → attach to response (non-pupil only)
  12. `res.status(201).json(result)` → response sent
  13. Non-blocking email to coordinators + head_teachers (if tier >= 2)
      → fire-and-forget IIFE after response

- **Pattern detection flow:**
  1. `runPatternDetection(incident)` called with single incident
  2. Checks sexual category → immediate red alert via createAlert()
  3. Checks repeat victim (same victim, 3+ incidents in 30 days) → amber
  4. Checks escalating severity (tier increase pattern) → amber
  5. Checks category frequency (3+ same category from same perpetrator
     in 30 days) → amber
  6. Each check calls `createAlert()` which deduplicates, inserts, and
     fires email for amber/red alerts
  7. `notifyCoordinators()` creates in-app notifications

- **Pupil login lockout flow:**
  1. Client POST /auth/pupil/login with loginSessionToken, loginKey, pin
  2. loginSessions Map looked up (in-memory — NOT persistent)
  3. User record fetched from DB (including lockedUntil,
     failedLoginAttempts)
  4. If lockedUntil > now → check if admin lock (> 24h from now) or
     timed lock → return 423 with appropriate message
  5. PIN compared via bcrypt
  6. If wrong: increment failedLoginAttempts. At 5 → admin lock
     (2099-12-31). At 3 → 15-minute lock. Below 3 → warn with remaining
     attempts.
  7. If correct: reset failedLoginAttempts and lockedUntil, issue JWT

- **Email delivery flow:**
  1. Caller invokes sendEmail() with opts
  2. getResend() returns cached Resend client (or null if no API key)
  3. If null → log warning, return silently
  4. Call client.emails.send() with from, to, subject, text
  5. If response has { error } → console.error + writeAudit
  6. If exception thrown → console.error + writeAudit
  7. Never rethrows — caller is never interrupted

- **PDF generation flow:**
  1. Route handler calls generateIncidentPdf() or generateProtocolPdf()
  2. PDFDocument created with A4 size, buffered pages
  3. Content written via addHeader, addSectionTitle, addField helpers
  4. Buffer collected via doc.on("data") chunk accumulation
  5. doc.on("end") resolves Promise<Buffer>
  6. Route handler sends buffer via res.send(buffer)

### 7. Patterns and conventions

- **Non-blocking email sends:** All email triggers fire after the HTTP
  response via `(async () => { ... })()` fire-and-forget IIFE pattern.
  Promise.all wraps individual sends. Outer catch(){} silences
  unhandled rejections. New email triggers must follow this pattern.

- **Audit-on-failure only:** sendEmail() audits only on failure
  (eventType "email_send_failed"). Successful sends are not audited.

- **Pattern detection deduplication:** createAlert() checks for existing
  open alert with same ruleId + schoolId + victimId before inserting.
  Prevents duplicate alerts.

- **In-app notification pattern:** Direct insert to notificationsTable
  with trigger identifier, channel "in_app", delivered: true. No
  notification service abstraction.

- **Escalation tier classification:** Pure function (no DB access).
  Category string → tier number. Tier 3 = sexual/coercive, Tier 2 =
  physical/psychological/online, Tier 1 = everything else.

- **Progressive lockout:** Timed lock (15 min) at 3 failed attempts,
  admin lock (2099-12-31) at 5 failed attempts. Admin lock requires
  coordinator PIN reset. Threshold constants at module level.

- **PDF generation:** Buffer-based (not stream). Collect chunks via
  event listener, resolve Promise<Buffer>. Route sends via
  res.send(buffer). Do not use stream piping.

- **In-memory state (auth.ts):** loginSessions Map is process-local.
  Known architectural weakness (FIX-04). Not shared across autoscale
  instances.

- **Route ordering:** Static routes declared before parameterised
  `:id` routes in same router to prevent matching conflicts.

- **Reference number format:** Incidents = SS-YYYY-NNNN, Protocols =
  PROT-YYYY-NNN.

- **Zod validation:** Request bodies parsed with .safeParse() (not
  .parse()). Error returns 400 with error message.

- **Auth middleware chain:** authMiddleware extracts JWT → requireRole()
  checks role(s). JwtPayload: { userId, schoolId, role, email }.

### 8. Inconsistencies and flags

- **In-app notification tier threshold mismatch:** In-app notifications
  to coordinators fire at `escalationTier >= 3 || safeguardingTrigger`
  (line 218 of incidents.ts), but email notifications fire at
  `escalationTier >= 2` (line 259). This means Tier 2 incidents
  (physical, psychological, online) trigger an email but NOT an in-app
  notification to coordinators. This is intentional per phase doc
  RR-2026-04-12-007 but may confuse coordinators who receive an email
  without a corresponding in-app notification.

- **loginSessions in-memory Map:** Not persistent, not shared across
  instances. Known FIX-04. If the process restarts during a pupil login
  flow, the session is lost. If multiple instances run, a pupil's login
  session exists only on the instance that created it.

- **Empty catch blocks:** Both email trigger IIFEs (incidents.ts line
  282, patternDetection.ts line 331) use `catch {}` which silently
  swallows errors. The individual sendEmail() calls within already
  handle their own errors, so this outer catch is a safety net, but it
  means any error in the email recipient lookup (DB query failure) would
  also be silently swallowed.

- **No test infrastructure:** Zero test coverage across the entire
  monorepo. No test runner installed, no test config files, no test
  scripts. Any code change relies entirely on manual testing and e2e
  Playwright runs (external to the codebase).

- **`incident` parameter typing in patternDetection.ts:**
  `runPatternDetection` accepts `typeof incidentsTable.$inferSelect`
  which is the full inferred row type. This couples pattern detection
  directly to the Drizzle schema. If the schema changes, the function
  signature auto-updates but callers may break silently.

- **Raw SQL in runMoodDeclineScan:** Uses `db.execute(sql\`...\`)` with
  raw column names (pupil_id, school_id, mood, created_at) and casts
  result rows as `any[]`. This bypasses Drizzle's type safety and will
  not catch column rename errors at compile time.
